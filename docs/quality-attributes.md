# 주요 품질속성 정리

본 문서는 `docs/requirements.md`의 Windows Image Viewer + Archive Browser 요구사항을 바탕으로 도출한 주요 품질속성이다. 각 품질속성은 요구사항 근거, 시스템에 미치는 의미, 검증 가능한 기준을 함께 정리한다.

## 1. 성능

### 요구사항 근거

- 대용량 이미지 데이터셋, 만화, 사진 아카이브 최적화
- 최소 지연 시간 목표: 사용자 인지 기준 100ms 미만
- 500MB 이하 ZIP 아카이브 열기 200ms 미만
- 일반 이미지 디코딩 100ms 미만
- 60 FPS 이상 탐색
- SSD 기준 콜드 스타트 1초 미만

### 품질속성 설명

이 애플리케이션의 핵심 가치는 이미지와 압축 파일을 즉시 열고 부드럽게 탐색하는 것이다. 따라서 성능은 가장 중요한 품질속성이다. 특히 압축 파일 내부 이미지를 추출 없이 열어야 하므로, 아카이브 인덱싱, 이미지 디코딩, 썸네일 생성, GPU 렌더링이 병목 없이 동작해야 한다.

### 품질 시나리오

- 사용자가 500MB 이하 ZIP 파일을 열면, 시스템은 200ms 이내에 탐색 가능한 목록을 표시한다.
- 사용자가 다음/이전 이미지로 이동하면, 일반 이미지 기준 100ms 이내에 표시된다.
- 사용자가 확대, 축소, 이동, 회전 작업을 수행해도 화면 전환은 최소 60 FPS를 유지한다.
- 사용자가 앱을 실행하면, SSD 환경에서 1초 이내에 기본 화면이 사용 가능해야 한다.

### 측정 기준

- 아카이브 오픈 시간
- 이미지 디코딩 시간
- 프레임 타임 및 FPS
- 콜드 스타트 시간
- 썸네일 생성 큐 지연 시간

## 2. 사용성

### 요구사항 근거

- 키보드 탐색, 마우스 휠 확대/탐색
- 전체화면 몰입 모드
- 최소 오버레이 UI
- 북마크/즐겨찾기
- 인덱스 사이드바
- 드래그 앤 드롭
- 멀티 모니터 지원
- Windows Explorer 컨텍스트 메뉴와 파일 연결

### 품질속성 설명

이미지 뷰어는 반복 사용 시간이 긴 도구이므로 조작이 직관적이고 빠르게 이어져야 한다. 로컬 파일과 아카이브 내부 파일의 차이를 사용자가 의식하지 않아도 되도록 UX가 통합되어야 하며, 키보드 중심 사용자와 마우스 중심 사용자 모두에게 자연스러운 흐름을 제공해야 한다.

### 품질 시나리오

- 사용자는 방향키와 PageUp/PageDown으로 이미지 목록을 끊김 없이 탐색한다.
- 사용자는 전체화면 상태에서도 최소한의 오버레이만 보고 이미지 감상을 방해받지 않는다.
- 사용자는 Explorer에서 이미지 또는 아카이브를 우클릭해 바로 뷰어로 열 수 있다.
- 사용자는 로컬 폴더와 압축 파일 내부를 동일한 방식으로 탐색한다.

### 측정 기준

- 주요 작업 완료까지 필요한 입력 횟수
- 키보드/마우스 단축 동작 응답 시간
- 전체화면 UI 가림 영역
- 사용자 테스트에서의 작업 성공률

## 3. 확장성

### 요구사항 근거

- 플러그인 아키텍처
- 아카이브 포맷, 이미지 코덱, 메타데이터 제공자 확장
- 동적 공유 라이브러리 기반 플러그인
- 버전 관리되는 플러그인 API
- 스크립터블 훅은 고급 기능으로 고려

### 품질속성 설명

이미지 포맷과 압축 포맷은 계속 추가될 수 있으므로, 핵심 엔진을 수정하지 않고 기능을 확장할 수 있어야 한다. 플러그인 API는 버전 호환성을 보장해야 하며, 코덱, 아카이브 엔진, 메타데이터 파서를 독립적으로 교체하거나 추가할 수 있어야 한다.

### 품질 시나리오

- 새로운 이미지 코덱 플러그인을 추가하면 기존 뷰어 기능을 수정하지 않고 새 포맷을 열 수 있다.
- 새로운 아카이브 포맷 지원 DLL을 배포하면 VFS 계층에서 동일한 탐색 방식으로 사용할 수 있다.
- 플러그인 API 버전이 맞지 않으면 앱은 충돌하지 않고 비활성화 또는 경고 처리한다.

### 측정 기준

- 신규 포맷 추가 시 수정해야 하는 핵심 코드 범위
- 플러그인 로딩 실패 시 앱 안정성
- 플러그인 API 하위 호환성 유지 범위

## 4. 보안

### 요구사항 근거

- Zip bomb 보호
- 경로 탐색 공격 방지
- 최대 파일 수, 최대 아카이브 크기 등 리소스 제한
- 샌드박스된 아카이브 파싱
- 손상 파일 처리

### 품질속성 설명

압축 파일과 이미지 파일은 외부 입력이며, 악성 파일이 포함될 수 있다. 특히 압축 해제 없이 내부를 탐색하는 기능은 Zip bomb, 경로 조작, 과도한 리소스 사용에 노출될 수 있다. 따라서 파일 파싱 계층은 반드시 방어적으로 설계되어야 한다.

### 품질 시나리오

- 압축률이 비정상적으로 높은 파일을 열면 시스템은 제한을 적용하고 처리를 중단한다.
- 아카이브 내부 경로에 `../`가 포함되어 있어도 로컬 파일시스템 외부로 접근하지 않는다.
- 파일 수나 크기가 제한을 초과하면 사용자에게 오류를 표시하고 앱은 정상 상태를 유지한다.
- 손상된 아카이브나 이미지가 입력되어도 앱은 충돌하지 않는다.

### 측정 기준

- Zip bomb 탐지 및 차단 여부
- 경로 정규화 테스트 통과 여부
- 리소스 제한 초과 시 메모리/CPU 안정성
- 악성/손상 샘플 처리 중 크래시 발생률

## 5. 신뢰성

### 요구사항 근거

- 손상된 아카이브 처리
- 부분 파일 복구
- 캐시 재빌드 모드
- 썸네일 비동기 생성 및 캐시
- LRU 기반 메모리 제한

### 품질속성 설명

앱은 다양한 상태의 파일을 다루므로 오류 상황에서도 계속 사용할 수 있어야 한다. 손상된 아카이브, 깨진 이미지, 캐시 불일치, 일부 파일 접근 실패 같은 상황을 정상적인 예외 흐름으로 다뤄야 한다.

### 품질 시나리오

- 썸네일 캐시가 손상되면 앱은 캐시를 재빌드하고 탐색 기능은 유지한다.
- 아카이브 내부 일부 이미지가 깨져 있어도 나머지 이미지는 계속 탐색 가능하다.
- 이미지 디코딩 실패 시 앱은 오류 표시 후 다음 파일 탐색을 허용한다.

### 측정 기준

- 손상 파일 테스트 케이스 통과율
- 캐시 재빌드 성공률
- 예외 상황에서 앱 크래시 발생률
- 실패한 항목 이후 탐색 지속 가능 여부

## 6. 자원 효율성

### 요구사항 근거

- 엄격한 메모리 제한과 LRU 제거
- 낮은 유휴 CPU 사용량
- 비동기 썸네일 생성
- 메모리 LRU 캐시와 디스크 영속 캐시
- 다음/이전 이미지 프리패치

### 품질속성 설명

대용량 폴더와 아카이브를 다루면 이미지, 썸네일, 메타데이터 캐시가 빠르게 증가한다. 성능을 높이기 위해 캐시와 프리패치를 사용하되, 메모리와 CPU를 통제해야 한다. 특히 백그라운드 스캐너와 썸네일 생성기는 사용자 조작을 방해하지 않아야 한다.

### 품질 시나리오

- 수천 장의 이미지가 있는 폴더를 탐색해도 메모리 사용량은 설정된 상한을 넘지 않는다.
- 사용자가 아무 작업을 하지 않는 상태에서는 CPU 사용률이 낮게 유지된다.
- 썸네일 생성 중에도 이미지 탐색과 확대/축소가 지연되지 않는다.

### 측정 기준

- 최대 메모리 사용량
- 유휴 상태 CPU 사용률
- 캐시 적중률
- 백그라운드 작업 중 UI 응답 시간

## 7. 이식성 및 배포성

### 요구사항 근거

- Windows 10/11 지원
- Installer mode와 Portable mode 지원
- Qt Installer Framework 권장
- 파일 연결, 레지스트리 설정, 셸 컨텍스트 메뉴 등록
- 설치 제거 시 정리

### 품질속성 설명

주 대상 플랫폼은 Windows이며, 일반 설치형 사용과 무설치 포터블 사용을 모두 지원해야 한다. 설치형 배포에서는 파일 연결과 셸 통합이 중요하고, 포터블 모드에서는 레지스트리 의존성을 최소화해야 한다.

### 품질 시나리오

- 사용자가 설치 모드로 배포하면 파일 연결과 컨텍스트 메뉴가 정상 등록된다.
- 사용자가 앱을 제거하면 설치 과정에서 등록한 항목이 정리된다.
- 사용자가 포터블 모드로 실행하면 관리자 권한이나 레지스트리 등록 없이 기본 이미지 탐색 기능을 사용할 수 있다.

### 측정 기준

- Windows 10/11 설치 및 실행 테스트 통과 여부
- 설치/제거 후 레지스트리와 파일 연결 상태
- 포터블 모드에서 기능 제한 범위

## 8. 유지보수성

### 요구사항 근거

- Qt UI Layer와 C++20 Core Engine 분리
- VFS, Archive Engine, Decode Pipeline, Thumbnail Pipeline의 모듈화
- CMake 필수
- vcpkg 또는 Conan 기반 의존성 관리
- 명확한 기술 스택 고정

### 품질속성 설명

요구사항은 UI, 코어 엔진, 아카이브 엔진, 디코더, VFS, 캐시 관리자 등을 분리하도록 명시한다. 이는 성능과 확장성뿐 아니라 유지보수성에도 중요하다. 각 모듈이 명확한 책임을 가지면 테스트와 교체가 쉬워지고, 플랫폼 통합 코드와 핵심 로직의 결합을 줄일 수 있다.

### 품질 시나리오

- VFS 계층을 수정하지 않고 UI 표시 방식을 변경할 수 있다.
- 특정 이미지 디코더를 교체해도 아카이브 탐색 로직에는 영향이 없다.
- CMake 기반 빌드로 개발 환경과 CI에서 동일한 빌드 절차를 수행한다.

### 측정 기준

- 모듈 간 의존성 방향 준수 여부
- 단위 테스트 가능한 코어 모듈 비율
- 신규 의존성 추가 시 빌드 설정 변경 범위
- UI 변경이 코어 엔진에 미치는 영향도

## 9. 상호운용성

### 요구사항 근거

- Windows Shell 통합
- 파일 연결
- Explorer 컨텍스트 메뉴
- COM 기반 셸 확장
- 썸네일 제공자와 미리보기 핸들러
- `dongviewer://open?path=...` 프로토콜 핸들러

### 품질속성 설명

앱은 독립 실행형 뷰어뿐 아니라 Windows 탐색기와 자연스럽게 연결되는 도구여야 한다. 사용자는 파일, 폴더, 아카이브, 프로토콜 호출 등 다양한 진입점에서 동일한 뷰어 경험을 얻어야 한다.

### 품질 시나리오

- 사용자가 Explorer에서 이미지 파일을 더블클릭하면 앱이 연결 프로그램으로 실행된다.
- 사용자가 아카이브 파일을 우클릭하면 Browse Archive 메뉴로 바로 탐색할 수 있다.
- 외부 프로그램이 `dongviewer://open?path=...`를 호출하면 해당 경로가 앱에서 열린다.

### 측정 기준

- 파일 연결 동작 테스트 통과 여부
- 컨텍스트 메뉴 등록 및 실행 성공률
- 셸 확장 크래시 격리 여부
- 프로토콜 핸들러 입력 검증 여부

## 우선순위 요약

| 우선순위 | 품질속성 | 이유 |
|---|---|---|
| 1 | 성능 | 제품의 핵심 목표가 즉시 표시, 빠른 탐색, 60 FPS 렌더링에 있음 |
| 2 | 사용성 | 이미지 감상과 아카이브 탐색의 반복 작업 효율이 중요함 |
| 3 | 보안 | 외부 압축 파일과 이미지 파일을 직접 파싱하므로 공격면이 큼 |
| 4 | 신뢰성 | 손상 파일, 캐시 문제, 디코딩 실패에도 앱이 계속 동작해야 함 |
| 5 | 자원 효율성 | 대용량 데이터셋 처리 시 메모리와 CPU 통제가 필수임 |
| 6 | 확장성 | 이미지/아카이브 포맷 추가를 플러그인으로 수용해야 함 |
| 7 | 유지보수성 | UI, VFS, 디코더, 아카이브 엔진 분리가 장기 개발에 중요함 |
| 8 | 배포성 | 설치형/포터블 모드와 Windows 통합을 안정적으로 제공해야 함 |

## 핵심 결론

이 요구사항에서 가장 중요한 품질속성은 성능, 사용성, 보안, 신뢰성이다. 특히 이 애플리케이션은 단순 이미지 뷰어가 아니라 압축 파일 내부를 추출 없이 탐색하는 구조이므로, VFS와 아카이브 파싱 계층의 성능과 안전성이 전체 품질을 좌우한다. 설계 단계에서는 비동기 파이프라인, GPU 렌더링, 캐시 정책, 리소스 제한, 오류 복구 흐름을 우선적으로 구체화해야 한다.
